# clipboardwire — wire protocol v0.1

This document specifies the protocol between a clipboardwire **client** and the
clipboardwire **server**. It is the only contract between the two halves; either
side can be reimplemented in any language as long as this document is honored.

Version: **0.1** (unstable; will break until 1.0).

## 1. Overview

- Transport: **WebSocket** over HTTP/1.1. Server speaks plain `ws://`; TLS is the
  reverse proxy's job.
- Authentication: **HTTP Basic** on the upgrade request. One user, configured in the
  server's environment.
- Confidentiality: **transport-only.** TLS at the reverse proxy protects the wire;
  the server itself sees clipboard contents in plaintext. This is by design — see
  the README's threat-model section. There is **no** client-side encryption layer.
- Topology: star. The server is a dumb hub that fans out frames to all
  authenticated peers except the sender. There is no persistent state on disk.

## 2. Connection lifecycle

### 2.1 Handshake

1. Client opens a WebSocket to `ws://<host>:<port>/sync`.
2. The HTTP upgrade request **must** carry an `Authorization: Basic <b64>` header,
   where `<b64>` is `base64(user + ":" + password)` per RFC 7617.
3. The server validates credentials using a constant-time comparison.
   - On failure the server responds with HTTP `401 Unauthorized` and the WebSocket
     is never opened. No body, no hints.
   - On success the server completes the upgrade (HTTP `101`).
4. Immediately after the upgrade succeeds, the server sends one `welcome` frame
   (see §3.2).

### 2.2 Steady state

After `welcome`, the connection is symmetric: either side may send frames at any
time. Frames are JSON-encoded WebSocket **text** frames, one JSON object per frame.

### 2.3 Keep-alive

The server sends a WebSocket **Ping** every 30 s. The client must reply **Pong**.
The server closes any connection that fails to pong within 10 s of a ping.
Clients should send Pings of their own if they don't see server traffic for 60 s
and reconnect (§2.5) if the server doesn't pong within 10 s.

### 2.4 Closure

Either side may close cleanly with code `1000`. Server-initiated error closures
use the codes in §6.

### 2.5 Reconnect

Clients must reconnect with exponential backoff: initial 1 s, doubling to a cap
of 60 s, with ±20 % jitter. The backoff resets to 1 s after a connection has been
open for at least 30 s.

## 3. Frame types

Every frame is a JSON object with a string field `type`. Unknown frame types are
ignored by both sides (forward-compat).

### 3.1 `clip` — clipboard payload

Sent by either side. The publishing client wraps a clipboard change; the server
relays the *same frame body* to every other connected peer, adding a `from` field.

```json
{
  "type": "clip",
  "id": "8c4a2e6e-7a3e-4d7c-9b1f-1d2e3f4a5b6c",
  "ts": 1716163200000,
  "content_type": "text/plain; charset=utf-8",
  "content": "the clipboard text, verbatim"
}
```

Field rules:

| Field          | Required        | Notes                                                                                                                  |
| -------------- | --------------- | ---------------------------------------------------------------------------------------------------------------------- |
| `id`           | yes             | RFC 4122 UUIDv4, generated by the publishing client. Used for dedup.                                                   |
| `ts`           | yes             | Unix epoch milliseconds at the publisher, used by clients to drop stale frames.                                        |
| `content_type` | yes             | MIME type of `content`. v0.1 supports only `text/plain; charset=utf-8`. Other values cause close `4004`.               |
| `content`      | yes             | The clipboard payload as a JSON string. v0.1: UTF-8 text only. Binary types will use a base64 sibling field in a future revision. |
| `from`         | added by server | Server-assigned per-connection client id (UUID string). Clients should ignore frames where `from` matches their own id. |

The server *may* log frame `id`, `ts`, `content_type`, length, and `from` for
observability. It **must not** log `content` at INFO level or above.

### 3.2 `welcome` — server greeting

Sent by the server exactly once, immediately after the upgrade.

```json
{
  "type": "welcome",
  "server": "clipboardwire/0.1.0",
  "client_id": "f0e9d8c7-b6a5-4321-9876-0123456789ab",
  "last_clip": null
}
```

- `client_id` — the id the server will tag this connection's `from` with on
  relayed frames. The client uses this to recognize its own echoes.
- `last_clip` — either `null` or a complete `clip` frame (the most recent clip
  the server has cached). The client may decide whether to apply it.

### 3.3 `error` — server-side soft error

```json
{ "type": "error", "code": "frame_too_large", "message": "max 10485760 bytes" }
```

Sent before a non-fatal disconnect, or to acknowledge a rejected frame without
closing the socket. Codes are the §6 strings.

## 4. Security model

clipboardwire v0.1 has **no application-layer cryptography**. Confidentiality and
integrity on the wire come entirely from TLS terminating at a reverse proxy
(Caddy / nginx / Traefik / Cloudflare Tunnel — operator's choice). The server
processes plaintext clipboard contents in memory; whether it logs them is
constrained by §3.1.

| Threat                                  | Mitigated?                                                            |
| --------------------------------------- | --------------------------------------------------------------------- |
| Network attacker reads clipboard        | yes, via TLS at the reverse proxy.                                    |
| Network attacker tampers with clipboard | yes, via TLS.                                                         |
| Server operator reads clipboard         | **no.** Out of threat model for v0.1.                                 |
| Server disk / RAM theft                 | **no.** `last_clip` lives in process memory; logs may contain frame metadata. |
| Replayed frames after reconnect         | mitigated: clients drop frames whose `ts` is older than the last applied `ts`. |
| Stolen Basic-auth credentials           | grants full read/write access; rotate the server password.            |
| Traffic analysis (sizes, timings)       | no.                                                                   |

This is appropriate for a LAN- or VPN-only deployment on hardware the user owns.
A future protocol revision may introduce optional end-to-end encryption with
TOFU device pairing (similar to Signal Desktop's QR-code flow), at which point
the `clip` frame will gain `nonce` / `ciphertext` fields and `content` will
become mutually exclusive with them. The wire-version negotiation in `welcome`
(see §7) is the upgrade hook.

## 5. Limits

- Max WebSocket frame size: **10 MiB** (10 × 1024 × 1024 bytes). Larger frames
  cause an `error` of code `frame_too_large` followed by close `4002`.
- Max concurrent connections per server: **64**. Exceeding this responds to the
  upgrade with HTTP `503`.
- Server `last_clip` cache: in-memory only; one slot, last-write-wins. Lost on
  process restart.

## 6. Close codes

The server uses these WebSocket close codes (private range 4000–4999) plus
standard `1000`/`1001`.

| Code   | Name              | Meaning                                                       |
| ------ | ----------------- | ------------------------------------------------------------- |
| `1000` | normal            | Graceful shutdown initiated by either side.                   |
| `1001` | going away        | Server is shutting down.                                      |
| `4001` | unauthorized      | Reserved; auth failure normally surfaces as HTTP 401 before upgrade. |
| `4002` | bad_frame         | Malformed JSON, missing required fields, or wrong field types. |
| `4003` | frame_too_large   | Frame exceeded §5 limit.                                      |
| `4004` | unsupported_type  | Frame `type` is known but not supported in this server build. |
| `4005` | duplicate_session | Reserved. Not enforced in v0.1.                               |

## 7. Versioning

This document is versioned in lockstep with the server. Breaking changes bump the
minor version while < 1.0. The server advertises its version in `welcome.server`;
clients should refuse to connect to a server whose advertised major.minor differs
from the one they were built for, and surface a clear error.

## 8. Out of scope (for v0.1)

- Multi-user support (would need a per-user namespace on the URL and per-user
  fan-out maps).
- Binary clipboard payloads (images, files).
- Push-style clipboard events on platforms that support them (we poll instead).
- File-based persistence for `last_clip`.
- Application-layer encryption / TOFU device pairing (see §4 for the future
  upgrade path).
